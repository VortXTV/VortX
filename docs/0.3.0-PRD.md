# StremioX 0.3.0 PRD — Native iOS/iPadOS + the cross-platform foundation

Status: in progress (started 2026-06-13). Built iteratively; everything ships as **prerelease** until validated on device. Grounded in the private competitor research ([[competitor-gaps]], [[official-v2-analysis]]) and the existing [REBASE-iOS.md](REBASE-iOS.md) plan.

## Goal

Make 0.3.0 a release worth the version bump: a **native iPhone + iPad app** on the same stremio-core + libmpv engine as the Apple TV app (retiring the web-host interim), shipping with a deliberate batch of the highest-leverage features from our competitor research, and laying the architectural groundwork so Mac, Android, Android TV, and a Windows/desktop build can follow on the shared core.

## Why now

- The engine layer (`SourcesShared`: CoreBridge, CoreModels, StreamRanking, Profiles, StremioAccount, ThemeManager, SubtitleAddons, SourcePreferences) and the libmpv player (`Sources/Player`) are already platform-agnostic and shipping on tvOS. iOS reuses them wholesale; only the UI layer is new.
- The official iOS app un-froze (v2.0.0) and added native downloads + profiles. Our tvOS edge holds (official TV is still Lite), but iOS parity now matters. Offline downloads is the new table-stakes gap.
- Most "top-pick" comp features are already shipped on tvOS (auto-failover/recovery, in-player source switcher, anti-cam/junk ranking, binge auto-next, per-profile languages, add-on subtitles). 0.3.0 brings them to iOS for free via the shared engine, plus adds the iOS-specific wins.

## Scope — what 0.3.0 ships

### Track 1: Native iOS/iPadOS app (the headline)
Follows [REBASE-iOS.md](REBASE-iOS.md). New `SourcesiOS` UI group; reuse `SourcesShared` + `Sources/Player`.
1. Engine for iOS: confirm `StremioXCore.xcframework` has the iOS device + sim slices; smoke-test `CoreBridge.shared.start()` natively.
2. App shell: bottom `TabView` (Home / Discover / Library / Search / Settings), `NavigationStack`, sign-in seeds the engine (reuse `StremioAccount`, QR + password).
3. Touch screens, in order: Home (poster rails + Continue Watching), Detail + Streams (ranked Watch Now, quality picker, source list), Discover, Library, Search, Add-ons, Settings. iPhone single-column; iPad wider grids + split where it helps; portrait + landscape; Dynamic Type; 44pt targets.
4. Native iOS player: libmpv core with touch controls (tap to toggle, scrub, gestures), live progress + resume via the engine, the same options panels (audio/subtitle/aspect/quality/settings) adapted to touch.
5. Cut over: once parity is reached, remove `ContentView`/`StremioWebView` and the live-web proxy in `NodeServer`.

### Track 2: iOS-specific features (mined from research, iOS-relevant)
- **Picture-in-picture** (iPhone/iPad) — libmpv → AVSampleBufferDisplayLayer or PiP-capable path.
- **Lock Screen / Control Center now-playing + remote controls** (MPNowPlayingInfoCenter / MPRemoteCommandCenter).
- **Two-way external-player handoff** (Omni): hand off to Infuse/VLC/etc. AND receive resume/scrobble back (x-success callback parse). Extends the handoff already built.
- **Copy stream / magnet link in player** (official v2): trivial, pairs with handoff + QR.
- **Sign in with Apple** (official v2): a frictionless second auth path beside QR + password.
- **Background audio** for the music/podcast-style use and to keep audio while multitasking.

### Track 3: Cross-cutting features (land on tvOS + iOS together via the shared layer)
Prioritized from the comp-research "top picks" that aren't shipped yet and fit the current architecture:
- **Config snapshot export/import** (Omni/Harbor): back up add-ons + preferences to a shareable file; first-run "import your existing setup" (paste an add-on config URL / list, carry debrid keys). High retention value; pairs with the migration story.
- **In-app add-on store** fed by stremio-addons.net (ratings, trending) — install without typing a manifest URL on a remote.
- **Catalog Groups** (Omni): merge multiple add-on catalogs under one Home shortcut; pick which metadata add-on drives Home.
- **RPDB poster rating overlays** + **Fanart.tv HD logos/art**: cheap metadata polish (URL-templated).
- **"This Week" upcoming-episodes rail** + release calendar (TMDB + Library + Trakt).
- **Seek-preview thumbnails (trickplay)** on the scrubber — premium player feel; heavier, may slip to 0.5.x per the roadmap.

Offline downloads (the official-v2 top gap) is a 0.7.0 epic per the roadmap (server.js 4.21.0 exposes `/downloader/*`); keep it sequenced there unless it proves quick.

## Scope — the other platforms (separate tracks, post-0.3.0-iOS or parallel)

These cannot share the SwiftUI UI; they share only the engine concept. Realistic sequencing:

- **macOS (closest, cheap):** a Mac target on the SAME SwiftUI codebase via Mac Catalyst (or a thin AppKit shell). Reuses `SourcesShared` + `Sources/Player` (MPVKit has a macOS slice) + the iOS UI adapted for pointer/keyboard + window resizing. Targets the iOS UI once Track 1 lands. → its own milestone after iOS parity.
- **Android + Android TV (Kotlin track):** stremio-core is Rust → compiles to Android via JNI (`aarch64-linux-android`, `armv7`, `x86_64`). New Kotlin/Jetpack-Compose app (phone + a Leanback/Compose-for-TV surface), a JNI bridge mirroring `CoreBridge`'s JSON contract, and a player (libmpv via `libmpv` NDK build, or ExoPlayer/libVLC fallback). Entirely separate repo/dir; reuses only the Rust core and the add-on/ranking logic (port `StreamRanking` to Kotlin or call into the Rust core). → its own track; scaffold the core JNI build first.
- **Windows / desktop (web track):** mirrors how Stremio itself does desktop — a shell (Tauri, Rust-native, lighter than Electron) wrapping a web UI + the node streaming server, OR a from-scratch native app. Tauri + stremio-web-style UI + the embedded server is the pragmatic path; reuses the Rust core and server.js. → its own track.

Decision: build **Apple-shared first** (iOS → iPad → Mac), since 60-70% of that work is already done and shared. Then stand up the **Android/Android-TV Kotlin track** (core JNI build → bridge → UI → player). Then the **Windows/desktop Tauri track**. Each is its own set of loop iterations and its own prerelease line.

## Sequencing for the build loop (one item per iteration, build + prerelease as we go)

1. **iOS engine smoke test** — verify the xcframework iOS slices, CoreBridge.start() on the iOS sim. (Track 1.1)
2. **iOS app shell + sign-in** (Track 1.2)
3. **iOS Home + Detail + Streams** (Track 1.3, first surfaces) → first iOS prerelease.
4. **iOS player (touch)** (Track 1.4)
5. **iOS Discover / Library / Search / Add-ons / Settings** (Track 1.3, remaining)
6. **Cut over from web-host** (Track 1.5) → 0.3.0 native iOS prerelease.
7. iOS-specific: PiP, now-playing/remote, copy-link, two-way handoff, Sign in with Apple (Track 2).
8. Cross-cutting: config snapshot import/export + first-run import, add-on store, catalog groups, poster/logo art, "This Week" rail (Track 3) — land on both tvOS and iOS.
9. **macOS target** (Catalyst on the iOS UI).
10. **Android/Android-TV**: core JNI build → bridge → Compose UI → player.
11. **Windows/desktop**: Tauri shell + web UI + embedded server.

## Constraints

- Everything stays **prerelease** until validated on device; promotion to Latest is the maintainer's call.
- No competitor app names in repo/public docs. Credit contributors by name. User preference is the top ranking key. (See working conventions.)
- Each iteration: implement → build both affected targets → cut/refresh the prerelease → note what's verified vs needs device testing.
